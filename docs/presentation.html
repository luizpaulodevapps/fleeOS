Modelo Financeiro

Firebase Blaze Plan

Plano serverless puramente elástico cobrado conforme o volume exato de requisições e armazenamento consumidos.

  • Custo de Entrada Zero: A faixa gratuita (Spark) cobre volumes substanciais antes da primeira cobrança (50k reads/dia e 50k MAUs de Auth grátis).
  • Escalabilidade Elástica: Sem necessidade de preocupações com provisionamento de CPU/disco para suportar conexões simultâneas.

Métricas de Preço

- Firestore Reads: $0.06 / 100k
- Firestore Writes: $0.18 / 100k
- Storage: $0.026 / GB-mês
- Bandwidth: $0.12 / GB de download

Riscos

Queries redundantes ou ineficientes em listagens repetidas de veículos por motoristas podem multiplicar os custos exponenciais.

Estrutura do Firestore

Armazenamento NoSQL de documentos. Como não há joins de tabelas (ex: puxar motoristas e veículos em um único SELECT relacional), é necessário duplicar dados para não estourar a quantidade de consultas por operação.

// Desnormalização necessária para otimizar reads:
{
  id: "contrato-1",
  tenantId: "sa-east-1",
  driverId: "motorista-5",
  driverName: "João Silva", // Duplicado
  vehiclePlate: "ABC-1234" // Duplicado
}
Modelo Financeiro

Supabase Pro Plan

Plano relacional baseado em instâncias dedicadas de PostgreSQL, cobrando assinatura fixa com amplos limites inclusos.

  • Cota Base Inclusa Generosa: Assinatura de $25 inclui 8 GB de banco, 100k MAUs de Auth, 100 GB de Storage e 50 GB de banda de rede.
  • Custos Excedentes Reduzidos: Valores de banda e armazenamento extra são mais baixos que no Firebase (banda extra $0.09/GB vs $0.12/GB).

Métricas de Preço

- Assinatura Base: $25.00 / mês
- Excesso de DB: $0.125 / GB
- Excesso Storage: $0.021 / GB
- Excesso Banda: $0.09 / GB de download

Riscos

Custo inicial obrigatório de $25 mesmo que o sistema de frotas esteja com tráfego zerado ou em ambientes de homologação.

Estrutura do Postgres

Banco de dados SQL relacional clássico. Permite resolver agregações complexas via Joins otimizados em uma única leitura atômica, diminuindo expressivamente o consumo de rede da web e mobile.

-- Queries limpas sem duplicação de dados:
SELECT c.id, c.data_inicio,
  d.nome AS motorista, v.placa AS veiculo
FROM contratos c
JOIN motoristas d ON c.motorista_id = d.id
JOIN veiculos v ON c.veiculo_id = v.id
WHERE c.tenant_id = 'sa-east-1';

Gatilhos de Migração

Grátis ➔ Pago: Quando migrar?

Como o FleetOS atualmente roda nas faixas gratuitas do Firebase e Supabase, o crescimento exige monitoramento constante dos limites.

  • Gargalo Firebase (Spark): A migração para o plano Blaze é obrigatória se o armazenamento de imagens de checkout exceder 5 GB ou se as consultas diárias passarem de 50.000 leituras (ocorre próximo a 50 frotas ativas).
  • Gargalo Supabase (Free): Exige migração para o plano Pro ($25/mês) caso o tamanho da base Postgres ultrapasse 500 MB (dados de rastreamento) ou o Storage de fotos exceder 1 GB (~250 fotos).

Riscos & Benefícios

Riscos do Atraso na Migração:

Downtime repentino e erros de gravação em check-ins de motoristas ao estourar limites do Firebase. Suspensão automática por inatividade de 7 dias ou estouro de cota no Supabase.

Benefícios do Upgrade:

Fim de restrições rígidas com escalabilidade elástica pay-as-you-go (Firebase) ou garantia de CPU dedicada, backups automáticos diários, suporte SLA e banco sempre ativo (Supabase).

Roadmap Tecnológico

Investimento em Desenvolvimento

O esforço de engenharia de software pendente para transformar a arquitetura atual do FleetOS em um produto comercializável.

  • Isolamento de Dados Multi-Tenant: Implementação estrita de regras PostgreSQL RLS ou refine completo das `firestore.rules` para impedir vazamento de fotos/dados entre frotas concorrentes.
  • Faturamento de Assinatura (Stripe): Desenvolvimento do fluxo de checkout do inquilino integrado com controle de ativação/bloqueio automático de frotas com base no plano escolhido.

Funcionalidades Críticas de Negócio

Resolução de Conflito de Sync:

Resolver divergências quando múltiplos motoristas atualizam ou dão baixa no checklist do mesmo veículo de forma offline e simultânea.

Separação de Ambientes de Produção:

Substituir todas as simulações e mocks em memória por conexões reais e isoladas de bancos de dados para homologação (Staging) e produção.

Recomendação Arquitetural

Veredito de Infraestrutura

Com base nas análises técnica e financeira do ecossistema FleetOS, definimos a seguinte recomendação:

  • Supabase como Escolha Definitiva: A natureza estritamente relacional do sistema de gestão de frotas e contratos torna o Postgres muito superior na manutenção do código de backend e custos à medida que o sistema ganha escala.
  • Custo de Escala do Firebase: Embora o Firebase seja ideal para o pontapé inicial (MVP) por possuir barreira de custo zero, a precificação por leitura/escrita e taxas MAU elevadas encarecem as operações no médio/longo prazo.
